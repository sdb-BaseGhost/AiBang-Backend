# AI Coding 实战方法论：从需求到交付的闭环工作流

> 脱胎于《如何搭建一个端到端业务需求专家 Agent》的工程实践，提炼为个人/小团队可直接执行的 AI 协作开发方法论。
>
> 核心理念：**AI 写代码已经不是瓶颈，瓶颈在代码之外——上下文组织、质量卡口、反馈回收。**

---

## 0. 你为什么会需要这份文档

用 AI 写代码时，最常见的失败模式不是"代码写得烂"，而是：

| 问题 | 典型表现 |
|------|----------|
| 上下文不足 | AI 不了解你的业务背景，写出来的代码方向偏了你还看不出来 |
| 没有明确边界 | 需求说"做个用户系统"，AI 自己脑补了一堆你不需要的功能 |
| 缺少验收标准 | 代码写完了，不知道算不算"做完了"，来回改 |
| 反馈没有沉淀 | 上次踩的坑、纠正过的问题，下次换个对话又犯一遍 |
| 质量全靠自觉 | "帮我写完跑一下就行"，没有测试、没有 review、没有门禁 |

这份文档把上述问题系统化解决。

## 1. 需求进入：先组织上下文，再动手

### 1.1 写一份最小 Requirements（需求规格）

在让 AI 写第一行代码之前，先写成一个 `requirements.md` 放在项目根目录：

```markdown
# 需求规格：[项目名]

## 目标
用一句话说清楚这个项目要解决什么问题。

## 目标行为（做什么）
- [ ] 行为 1：...
- [ ] 行为 2：...
- [ ] 行为 3：...

## 非目标边界（不做什么）
- 不做 X（原因：...）
- 不做 Y（原因：...）

## 验收标准（怎么算做完了）
- 正例：核心场景 A 输入 xxx，输出 yyy
- 正例：核心场景 B 输入 xxx，输出 yyy
- 反例：边界条件 C 应该返回错误提示
- 反例：异常输入 D 应该被拒绝

## 技术约束
- 语言/框架：...
- 依赖服务：...
- 不可变约束：...

## 风险与待确认
- 问题 1：...
- 问题 2：...
```

**关键原则**：requirements 是你和 AI 的"合同"。AI 后续所有行为都以此为基准。如果你自己都说不清楚要做什么，AI 只会越努力越偏。

### 1.2 准备上下文快照

把 AI 需要知道的背景信息组织好，放在 `context/` 目录或直接写在项目说明里：

| 上下文类型 | 放哪里 | 示例 |
|-----------|--------|------|
| 业务背景 | `context/business.md` | 这个系统服务谁、解决什么问题、有哪些业务规则 |
| 技术现状 | `context/tech.md` | 现有代码结构、已有的模块、不能动的部分 |
| 参考资料 | `context/references.md` | API 文档链接、设计稿、竞品链接 |
| 历史决策 | `context/decisions.md` | 为什么选了这个框架、为什么不用某个方案 |

**不需要事无巨细**，但要保证 AI 在开始工作前能回答："这个系统是做什么的、现在长什么样、这次要改什么。"

### 1.3 建立项目记忆仓库（可选但推荐）

用一个目录专门存放项目记忆：

```
project-memory/
├── requirements.md        # 需求规格（每轮更新）
├── plan.md                # 技术方案
├── progress.md            # 进度记录
├── decisions.md           # 决策记录
├── feedback.md            # 反馈记录（AI 犯的错、你纠正过的问题）
└── closeout.md            # 结项总结
```

这些文件随着项目推进持续积累，下次开新需求时，AI 可以直接读取历史上下文。

## 2. 技术方案：先设计再编码（第二质量门）

### 2.1 写一份技术方案

Requirements 确认后，不要直接让 AI 写代码。先输出一份技术方案 `plan.md`：

```markdown
# 技术方案：[项目名]

## 现状
现在系统是什么样的。

## 目标
这次改动后系统应该变成什么样。

## 核心改动点
1. 改动 A：[描述] -> 影响范围：[模块 X]
2. 改动 B：[描述] -> 影响范围：[模块 Y]

## 不做什么
- 不改动 Z（原因：...）

## 风险
- 风险 1：[描述] -> 缓解措施：[...]

## 验收方式（这部分最重要！）
- 测试策略：单元测试覆盖 / 集成测试 / 手动验证
- 正例验证：[具体步骤]
- 反例验证：[具体步骤]
- 回归验证：[不应变化的行为]
```

### 2.2 关键原则：方案阶段就定好"怎么验收"

> 很多项目到了写完代码才开始想怎么验证，结果发现验收条件不明确、边界场景没覆盖，又要回头补。

**规则：没有验收标准的方案，不允许进入编码。**

验收标准必须是具体的、可执行的：
- "系统应该能正常工作" -- 模糊，不行
- "调用 /api/users?id=123 应返回 200 + 包含 name 字段的 JSON" -- 具体，可以
- "调用 /api/users?id=不存在 应返回 404 + 错误信息" -- 反例，也要写

## 3. 实现阶段：TDD + 质量门禁

### 3.1 用 TDD 驱动实现（不是先写代码再补测试）

这是整个方法论中**最容易被忽视、但价值最高**的一条。

**错误做法**：
1. 让 AI 写完所有代码
2. 然后说"帮我补一下测试"
3. AI 写出来的测试只是为了确认自己的代码能跑通（确认偏误）

**正确做法**：
1. 先根据 plan.md 里的验收标准，让 AI 写测试（测试定义"什么是对的"）
2. 测试写完后 review：测试是否覆盖了正例、反例、边界？
3. 确认测试没问题后，再让 AI 写实现代码（让代码去满足测试）
4. 跑测试，全部通过才算实现完成

```
流程：验收标准 -> 测试 -> Review 测试 -> 实现 -> 跑测试 -> 通过
```

### 3.2 设置 Pre-push 质量门禁（硬规则，不是提示词）

> 提示词再怎么写"push 前必须跑测试"，AI 在复杂任务中还是可能跳过。

**把质量门禁变成绕不过去的硬规则**：

#### Git Pre-push Hook 示例

```bash
#!/bin/bash
# .git/hooks/pre-push

echo "Running quality gate..."

# 1. 运行测试
echo "Running tests..."
npm test || { echo "Tests failed. Push rejected."; exit 1; }

# 2. 运行 lint
echo "Running lint..."
npm run lint || { echo "Lint errors found. Push rejected."; exit 1; }

# 3. 检查测试覆盖率（可选）
echo "Checking coverage..."
npm run coverage -- --threshold 70 || { echo "Coverage below 70%. Push rejected."; exit 1; }

echo "Quality gate passed."
```

### 3.3 Agent 内部 Review（自查清单）

每次让 AI 完成一段功能后，用以下清单自查：

**功能正确性**
- 核心场景按预期工作（跑正例测试）
- 边界条件被正确处理（跑反例测试）
- 没有引入回归问题（跑已有测试）

**代码质量**
- 没有 lint 错误
- 没有硬编码的魔法值
- 错误处理完整（不是只处理 happy path）
- 没有明显的安全问题（SQL 注入、XSS 等）

**可维护性**
- 函数/模块职责单一
- 命名清晰，不需要额外注释才能看懂
- 复杂逻辑有注释说明 why（不是 what）

## 4. 协作留痕：让反馈不蒸发

### 4.1 每次纠正 AI 都记录下来

当 AI 写的代码有问题、你纠正了它之后，把这次纠正记录到 `feedback.md`：

```markdown
## 反馈记录

### 2026-06-15：数据库查询缺少索引
- **场景**：AI 生成的查询没有加索引，在数据量大时会很慢
- **纠正**：要求 AI 在 WHERE 条件涉及的字段上加索引
- **沉淀**：以后让 AI 写 SQL 时，提醒它"检查是否需要索引"

### 2026-06-15：错误处理只覆盖 happy path
- **场景**：AI 写的 API 只处理了正常请求，没处理参数缺失、权限不足等情况
- **纠正**：要求 AI 补充完整的错误处理
- **沉淀**：以后写 API 时，提醒它"先定义错误响应格式，再写正常逻辑"
```

### 4.2 为什么这样做？

| 不记录 | 记录 |
|--------|------|
| 下次换个 AI 对话，同样的错误再犯 | 把 feedback.md 作为上下文喂给 AI，避免重蹈覆辙 |
| 纠正的经验只在你脑子里 | 经验可以共享给团队、复用到下一个项目 |
| 每次都要从头解释业务背景 | 历史反馈本身就是业务知识的一部分 |

## 5. 验收验证：用真实证据证明"做完了"

### 5.1 验收三件套

验收不是"跑一下看看没报错就行"。按以下三类逐项核对：

| 类型 | 问题 | 示例 |
|------|------|------|
| **正例** | 核心业务场景是否按预期工作？ | 用户注册 -> 收到确认邮件 -> 能登录 |
| **反例** | 边界条件和异常路径是否被正确处理？ | 重复注册 -> 返回"用户已存在" |
| **回归** | 改动是否影响了不应变化的行为？ | 改了登录逻辑 -> 但注册流程仍然正常 |

### 5.2 验收证据可回溯

每次验收后，把证据记录下来：

```markdown
## 验收记录

### 功能：用户注册
- 正例：POST /api/register -> 201 OK
- 反例：POST /api/register (空密码) -> 400 "密码不能为空"
- 回归：登录功能（POST /api/login）-> 仍然正常
- 覆盖率：注册模块 85%
```

### 5.3 自动化验收脚本（推荐）

把验收步骤写成可重复执行的脚本：

```bash
#!/bin/bash
# verify.sh - 验收脚本

echo "=== 开始验收 ==="

# 正例
echo "正例：核心功能测试"
npm test -- --grep "核心功能"

# 反例
echo "反例：边界条件测试"
npm test -- --grep "边界条件"

# 回归
echo "回归：已有功能测试"
npm test

# 覆盖率
echo "覆盖率检查"
npm run coverage

echo "=== 验收完成 ==="
```

## 6. 结项沉淀：让下个项目更轻松

### 6.1 结项时回答三个问题

项目完成后（或一个阶段完成后），花 10 分钟写一份 `closeout.md`：

```markdown
## 结项总结：[项目名]

### 1. 哪些知识是稳定的（下次直接复用）
- 业务规则：[...]
- 技术方案：[...]
- 验收规范：[...]

### 2. 哪些流程可以改进（下次避免踩坑）
- 问题 1：[AI 在 XX 环节犯了 YY 错误] -> 改进：[下次提醒 AI ZZ]
- 问题 2：[需求理解有偏差] -> 改进：[requirements 要更具体]

### 3. 哪些材料只需归档（不需要带到下个项目）
- 临时调试记录
- 被否决的方案
- 中间状态文件
```

### 6.2 形成你的 Skill / Prompt 库

把反复有效的经验变成可复用的 prompt 模板或 Claude Code Skill：

```
project-root/
├── .claude/
│   └── skills/
│       └── api-development.md    # API 开发的通用规范和流程
│       └── database-migration.md  # 数据库迁移的注意事项
│       └── testing-patterns.md    # 测试编写的模式和要求
└── ...
```

**示例 Skill** (`.claude/skills/api-development.md`)：

```markdown
---
name: api-development
description: 开发新的 API 端点时遵循的规范
---

# API 开发规范

## 开发流程
1. 先写 requirements（输入、输出、错误情况）
2. 先写测试（正例、反例、边界）
3. 再写实现
4. 运行测试通过

## 代码规范
- 所有 API 必须有输入验证
- 错误响应统一格式：{error: string, code: number}
- 必须处理：参数缺失、权限不足、资源不存在
- 数据库查询必须检查是否需要索引

## 验收标准
- 测试覆盖率 >= 70%
- 无 lint 错误
- 所有错误路径都有测试
```

## 7. 完整工作流速查表

```
+-----------------------------------------------------------------+
|                    AI Coding 工作流速查                          |
+-----------------------------------------------------------------+
|                                                                 |
|  1 需求进入                                                      |
|     +-- 写 requirements.md（目标、做什么、不做什么、验收标准）        |
|     +-- 准备上下文快照（业务背景、技术现状、参考资料）                  |
|     +-- 自检：requirements 是否清晰到 AI 能独立执行？                |
|                                                                 |
|  2 技术方案                                                      |
|     +-- 写 plan.md（现状、目标、改动点、风险、验收方式）               |
|     +-- 质量门：方案是否包含具体的验收标准？                          |
|     +-- 人工确认：方案没问题才进入编码                               |
|                                                                 |
|  3 实现                                                          |
|     +-- 先写测试（根据验收标准），再写实现                            |
|     +-- 自查清单：功能、质量、可维护性                               |
|     +-- PMD/lint 检查 -> 不过不提交                                |
|     +-- 质量门：测试全过 + lint 无错才 push                         |
|                                                                 |
|  4 验收                                                          |
|     +-- 正例：核心场景 OK                                          |
|     +-- 反例：边界条件 OK                                          |
|     +-- 回归：已有功能 OK                                          |
|     +-- 记录验收证据                                              |
|                                                                 |
|  5 反馈沉淀                                                      |
|     +-- 记录 AI 犯的错和你纠正的方式                                 |
|     +-- 更新 feedback.md                                         |
|     +-- 形成可复用的 skill / prompt 模板                            |
|                                                                 |
|  6 结项                                                          |
|     +-- 稳定知识 -> 长期 wiki / skill 库                          |
|     +-- 流程改进 -> 更新 prompt / skill                           |
|     +-- 一次性材料 -> 归档                                        |
|                                                                 |
+-----------------------------------------------------------------+
```

## 8. 给 Claude Code 用户的特别建议

如果你用 Claude Code 进行开发，以下是一些可以直接落地的实践：

### 8.1 用 CLAUDE.md 定义项目规范

在项目根目录创建 `CLAUDE.md`，把项目规范写进去：

```markdown
# 项目规范

## 开发流程
- 所有功能必须先写测试，再写实现
- 每次 commit 前必须通过 lint 和测试
- 不允许硬编码配置值

## 代码风格
- 使用 TypeScript strict mode
- 函数命名用 camelCase
- 错误处理必须覆盖所有外部调用

## 验收标准模板
- 每个 API 端点至少 3 个测试：正常调用、参数错误、权限不足
- 测试覆盖率 >= 70%

## 禁止事项
- 不要使用 any 类型
- 不要跳过错误处理
- 不要在没有测试的情况下提交代码
```

### 8.2 用 Hooks 卡质量门禁

利用 Claude Code 的 Hooks 机制，在工具调用前后自动执行检查。详见 [Claude Code Hooks 文档](https://code.claude.com/docs/en/hooks)。

### 8.3 用 Subagents 做 Code Review

让一个子 Agent 专门负责 review 主 Agent 写的代码：

```
主 Agent（写代码）-> 子 Agent（review）-> 主 Agent（修复）-> 子 Agent（确认）
```

### 8.4 上下文管理

- 定期用 `/compact` 压缩上下文，避免上下文膨胀导致质量下降
- 把重要信息放在 `CLAUDE.md` 或 `context/` 目录，而不是依赖对话历史
- 需求变更时，先更新 `requirements.md`，再让 AI 重新读取

## 9. 常见误区

| 误区 | 正确做法 |
|------|----------|
| "AI 那么聪明，直接说需求就行" | 写清楚 requirements，AI 不会读心术 |
| "先写完代码再说测试的事" | 先写测试定义"什么是对的"，再写代码 |
| "跑一下没报错就上线了" | 按正例、反例、回归三类逐项验收 |
| "上次的教训我记住了" | 写下来，下次把 feedback.md 喂给 AI |
| "prompt 写得好就行" | 提示词约束不如工程化卡口（hook / lint / test）可靠 |
| "AI 一次就能写对" | AI 写代码的正确姿势是迭代：写 -> 测试 -> 修 -> 再测 |
| "项目做完了就不管了" | 花 10 分钟做结项沉淀，下次能省 2 小时 |

---

## 10. 核心理念：一句话总结

> **上下文先行、方案先行、测试先行；质量门禁靠工程化不靠自觉；反馈留痕不蒸发；结项沉淀形成复利。**

这不是一套沉重的流程。它的核心是把"人反复补位"的动作，变成"系统自动承接"的机制。一开始可能觉得多写个 requirements.md 很麻烦，但当你发现 AI 第一次就写对了方向、测试一次就通过了、下次类似需求不用重新解释业务背景时，你就知道这些前置动作值不值了。
