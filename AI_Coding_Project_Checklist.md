# AI Coding 新项目启动检查清单

> 每次开新项目，按这份清单走一遍。不需要一次全做，但**前三项是底线**。
>
> 核心原则：**模型供给智商，框架供给纪律。**

---

## 第一步：项目骨架（30 分钟）

### 1.1 创建根 CLAUDE.md（≤ 8K tokens）

```markdown
# 项目定位
一句话说清楚这个项目是什么、解决什么问题。

# 技术栈
- 语言/框架：...
- 包管理：...
- 测试框架：...
- 部署方式：...

# 代码风格
- 命名规范：...
- 文件组织：...
- 禁止事项：（如：不要用 any、不要跳过错误处理、不要硬编码配置）

# 流程裁剪规则
- QUERY（纯查询）：直接回答，不启动开发流程
- TRIVIAL（改文案/修小 bug）：编码 → 自测 → 提交
- FEATURE（新功能）：需求 → 设计 → 测试先行 → 编码 → 门禁 → 验收
- ARCHITECTURE（架构改动）：需求 → 方案评审 → Pre-Mortem → TDD → 全链路验证

# 硬规则
- 先写测试，后写代码（TDD 顺序不可跳过）
- 测试必须经人工 review 后才允许写实现
- 改完必验证：检测到业务代码改动，必须跑完门禁才能提交
- 禁止危险操作：git push --force、rm -rf 等需人工确认

# 门禁速查
- G1: lint 通过 → 失败退回修改
- G2: 测试通过 → 失败退回开发
- G3: 覆盖率达标 → 失败退回补充测试
```

- [ ] 根 CLAUDE.md 已创建，且不超过 8K tokens
- [ ] 内容只包含：项目定位 + 技术栈 + 代码风格 + 流程裁剪 + 硬规则 + 门禁速查
- [ ] 没有把深度内容塞进来（TDD 指南、设计模板等放 context/ 层）

### 1.2 创建项目目录结构

```
my-project/
├── CLAUDE.md                  # 根入口（≤8K）
├── requirements.md            # 当前需求合同
├── design.md                  # 当前技术方案
├── feedback.md                # AI 纠错记录（实时维护）
├── state.json                 # 流程状态（长任务用）
├── rules/                     # 原子规则（踩坑沉淀）
│   ├── build.md
│   ├── test-first.md
│   └── commit-atomic.md
├── context/                   # 按需加载的深度内容
│   ├── tdd-guide.md
│   ├── pre-mortem-template.md
│   ├── adr-template.md
│   └── architecture.md
├── evals/                     # 门禁脚本
│   ├── gate-lint.sh
│   ├── gate-test.sh
│   └── gate-coverage.sh
├── modules/                   # 各功能模块
│   ├── module-a/
│   │   └── CLAUDE.md          # 模块级规范 + skill
│   └── module-b/
│       └── CLAUDE.md
└── .git/hooks/
    └── pre-push               # 硬门禁（lint + test 必过才能 push）
```

- [ ] 目录结构已创建
- [ ] rules/ 至少写 3 条你最痛的规则（从过去的坑中提炼）
- [ ] feedback.md 已创建（空文件也行，后面会持续填）

### 1.3 配置 pre-push hook（硬门禁）

```bash
#!/bin/bash
# .git/hooks/pre-push

echo "=== 质量门禁 ==="

echo "[G1] Lint 检查..."
npm run lint || { echo "❌ Lint 不通过，禁止 push"; exit 1; }

echo "[G2] 测试检查..."
npm test || { echo "❌ 测试不通过，禁止 push"; exit 1; }

echo "[G3] 覆盖率检查..."
npm run coverage -- --threshold 70 || { echo "❌ 覆盖率低于 70%，禁止 push"; exit 1; }

echo "✅ 门禁全部通过"
```

- [ ] pre-push hook 已配置且有执行权限
- [ ] 手动验证：故意写一个 lint 错误，确认 push 被拦住

## 第二步：每个需求的开发流程（按意图裁剪）

### 判断：这次任务走哪条路？

```
你的任务是什么？
│
├─ 只是问 AI 一个代码问题 ──────────────→ 直接回答，不启动流程
│
├─ 改个文案 / 修个小 bug ──────────────→ 快速路径
│                                        编码 → 自测 → 提交
│
├─ 开发新功能模块 ─────────────────────→ 标准路径（见下方完整流程）
│
└─ 架构级改动 / 跨模块重构 ────────────→ 全路径
                                         在标准路径前加：方案评审 + Pre-Mortem
```

### 标准路径完整流程

#### 2.1 写需求合同（requirements.md）

```markdown
# 需求：[功能名]

## 目标（一句话）
这个功能要解决什么问题。

## 做什么
- [ ] 具体行为 1
- [ ] 具体行为 2
- [ ] 具体行为 3

## 不做什么（明确边界，防止 AI 脑补）
- 不做 X（原因：...）

## 验收标准
- 正例：输入 xxx → 输出 yyy
- 正例：输入 xxx → 输出 yyy
- 反例：空输入 → 返回错误提示
- 反例：越权操作 → 返回 403
- 边界：空列表 / 超大输入 / 并发场景 → 预期行为

## 技术约束
- 必须使用现有 XX 模块，不能重写
- 数据库字段不能改动已有表结构
```

- [ ] requirements.md 已写好
- [ ] 验收标准是**具体可执行的**（不是"系统应该能正常工作"）
- [ ] 明确写了"不做什么"（防止 AI 范围蔓延）

#### 2.2 写技术方案（design.md）

```markdown
# 技术方案：[功能名]

## 现状
现在系统长什么样。

## 目标
改完后应该变成什么样。

## 核心改动点
1. 改动 A → 影响模块 X
2. 改动 B → 影响模块 Y

## 不做什么
- 不改动 Z（原因：...）

## 风险
- 风险 1 → 缓解措施：...
```

- [ ] design.md 已写好
- [ ] 已和 AI 沟通方案细节，AI 理解了改动范围

#### 2.3 Pre-Mortem（事前验尸，2 分钟）

让 AI 回答：

> "假设这个方案上线后出了严重问题，最可能的 3 个失败原因是什么？针对每个原因，当前方案中是否已有缓解措施？"

- [ ] 已做 Pre-Mortem
- [ ] 发现的风险已补充到 design.md 的风险章节

#### 2.4 TDD：先写测试，再写代码 ⚠️ 这步最关键

```
顺序（不可调换）：
  ① AI 根据验收标准，输出"测试要点清单"
  ② 你 review 测试要点 → 确认覆盖了正例/反例/边界
  ③ AI 写测试代码
  ④ 你 review 测试代码 → 确认测试本身是正确的
  ⑤ AI 写实现代码
  ⑥ 跑测试，全部通过
```

**review 测试时检查清单**：

- [ ] 正例：核心业务场景的正确路径有测试
- [ ] 反例：空值、非法输入、越权等错误路径有测试
- [ ] 边界：空列表、超大输入、并发等边界条件有测试
- [ ] 测试是为**需求**写的，不是为**实现**写的（换了实现方式测试也应该能跑通）

**实现完成后检查清单**：

- [ ] 所有测试通过
- [ ] 没有硬编码的魔法值
- [ ] 错误处理覆盖了所有外部调用（不只是 happy path）
- [ ] 没有 lint 错误
- [ ] 函数/模块职责单一，命名清晰

#### 2.5 门禁检查（自动 + 人工）

```
自动门禁（pre-push hook 自动拦截）：
  ✅ Lint 通过
  ✅ 测试通过
  ✅ 覆盖率达标

人工检查（你亲自看）：
  ✅ 代码改了什么，和设计文档一致吗
  ✅ 测试覆盖了 requirements 里的所有验收标准吗
  ✅ 有没有 AI 脑补的多余功能
```

- [ ] pre-push hook 运行通过
- [ ] 人工 review 确认改动与设计一致

#### 2.6 记录反馈（feedback.md）

**每次你纠正 AI，立刻记一笔**：

```markdown
### 2026-06-16：[简要描述]
- **场景**：AI 在什么情况下犯了什么错
- **纠正**：你怎么纠正的
- **沉淀**：以后遇到类似情况应该提醒 AI 什么
```

- [ ] 本次有无纠正 AI？如有，已记录到 feedback.md
- [ ] 纠正内容是否值得沉淀为 rule？如果是 → 加到 rules/ 目录

## 第三步：批量任务完成后（每 1~2 周一次）

### 3.1 经验进化

翻看 feedback.md，把积累的经验做分级：

| 级别 | 条件 | 动作 |
|------|------|------|
| **lesson** | 刚发生，只记录了一次 | 留在 feedback.md 里观察 |
| **pattern** | 同一个坑在不同场景出现 2~3 次 | 提炼为 rule 文件放到 rules/ |
| **instinct** | 验证过 3 次以上，确认规则可靠 | 注入根 CLAUDE.md 或 rules/ 成为硬规则 |

- [ ] 本轮 feedback.md 中的 lesson 是否有升级为 pattern 的
- [ ] 已有的 pattern 是否有升级为 instinct 的
- [ ] instinct 级规则是否已注入到 CLAUDE.md 或 rules/

### 3.2 结项沉淀

```markdown
## 本轮总结

### 稳定知识（下次直接复用）
- 业务规则：...
- 技术方案：...

### 流程改进（下次避免踩坑）
- 问题：... → 改进：...

### 一次性材料（归档不带走）
- 临时调试记录
- 被否决的方案
```

- [ ] 本轮结项总结已写
- [ ] 稳定知识已沉淀到 context/ 或 skill 中
- [ ] 流程改进已更新到 rules/ 或 CLAUDE.md

### 3.3 健康度审计

定期检查你的 harness 配置是否还健康：

- [ ] 根 CLAUDE.md 是否超过 8K tokens（超了就拆到 context/）
- [ ] rules/ 里的规则是否有过时的（事故场景已不存在）
- [ ] 门禁脚本是否还在正常工作
- [ ] 有没有新的高频坑需要新增规则

## 速查：一次完整需求的时间分配

```
写需求合同 (requirements.md)       5 min  ← 不能省，这是你和 AI 的合同
写技术方案 (design.md)             5 min  ← 不能省，防止 AI 想怎么写就怎么写
Pre-Mortem                         2 min  ← 推荐，提前暴露设计盲区
AI 写测试要点 → 你 review           3 min  ← 不能省，TDD 顺序是核心
AI 写测试代码 → 你 review           5 min  ← 不能省，防止确认偏误
AI 写实现代码                      10 min  ← 这步最快，因为测试已经定义了"对"
跑门禁 + 人工 review               3 min  ← 不能省，这是最后的安全网
记录 feedback（如有纠正）           1 min  ← 有就记，没有就跳过
────────────────────────────────────
合计约 30 min / 一个中等功能
```

> 你可能会觉得"以前直接让 AI 写只要 5 分钟"。但那 5 分钟省下来的时间，后面会以 3 倍的返工代价还回去。这 30 分钟是**一次做对**的成本，不是额外开销。
